# Informe de enfoque académico - Presentación NANU TECH

## 1. Ajustes de enfoque académico realizados

- Se reforzó que la presentación no es una demo aislada, sino una defensa de aplicación del curso Diseño e Implementación de Sistemas.
- Se agregó una frase guía en la diapositiva de objetivos para conectar el producto con análisis de negocio, SDLC, requerimientos, priorización, diseño, implementación, pruebas, despliegue y mantenimiento.
- Se incorporó un rótulo discreto por diapositiva con el concepto DISI aplicado.
- Se agregaron frases guía internas mediante atributos `data-speaker-note` para orientar la exposición sin reactivar un panel de notas ni ocupar espacio permanente.
- Se mantuvo el orden de las 28 diapositivas y las evidencias visuales existentes.

## 2. Diapositivas reforzadas por bloque

- Diapositivas 1-2: enfoque general, objetivos, alcance y ciclo completo del curso.
- Diapositivas 3-7: análisis de negocio, problema, segmento objetivo, justificación, idea TIC, Canvas, benchmark y propuesta de valor.
- Diapositivas 8-12: delimitación del MVP, supuestos, actores, canales y viabilidad inicial.
- Diapositivas 13-17: SDLC, metodología, ingeniería de requerimientos, reglas de negocio, MoSCoW y trazabilidad.
- Diapositivas 18-21: diseño funcional, UX/Figma, arquitectura, sincronización offline y modelo de datos.
- Diapositivas 22-24: implementación web, app móvil, backend/API, seguridad y auditoría.
- Diapositivas 25-27: pruebas, despliegue, costos, operación, capacitación y mantenimiento.
- Diapositiva 28: resultados, discusión crítica, conclusiones, límites y evolución futura.

## 3. Alineación con el informe final

La presentación queda alineada con el índice del informe final:

- Problema e introducción.
- Idea de negocio TIC.
- Marco aplicado: SDLC, requerimientos y priorización.
- Metodología.
- Análisis, diseño, implementación, pruebas, despliegue y mantenimiento.
- Resultados, discusión, conclusiones, recomendaciones y anexos.

## 4. Coherencia con el MVP

Se mantiene como incluido:

- Web administrativa y gerencial.
- Panel web del chofer como respaldo.
- App móvil del chofer como canal principal.
- Backend/API.
- Autenticación y roles.
- Gestión básica de camiones, contratos, jornadas, observaciones y alertas/SOS/auxilio.
- Auditoría.
- PostgreSQL y SQLite local móvil.
- Offline básico y sincronización posterior controlada.
- Pruebas funcionales e integración.
- Preparación local/testing/cloud inicial.

Se mantiene como excluido o futuro:

- GPS en tiempo real.
- Tracking en mapa vivo.
- Proveedor GPS externo.
- Combustible.
- Facturación.
- Mantenimiento avanzado.
- Push notifications.
- BI avanzado.
- Publicación en tiendas.
- Producción AWS estable garantizada.
- Offline robusto productivo.
- Sincronización background completa garantizada.

## 6. AWS y offline delimitados

- AWS se presenta como preparado para local/testing/cloud inicial, no como producción estable garantizada.
- Offline se presenta como básico/controlado, con cola local y sincronización posterior, no como offline robusto productivo.

## 7. Recomendaciones para exposición oral por bloques

- Inicio: enfatizar que el objetivo es demostrar aplicación del curso mediante un MVP.
- Negocio: explicar problema, segmento, Canvas y valor sin entrar aún en detalle técnico.
- Alcance: defender por qué el MVP incluye unas funciones y excluye otras.
- SDLC y análisis: explicar cómo se pasa de problema a requerimientos verificables.
- Diseño: conectar requerimientos con Figma, arquitectura, offline y datos.
- Implementación: mostrar web, app y backend como materialización del diseño.
- Pruebas y operación: defender evidencia QA, costos, despliegue inicial y mantenimiento.
- Cierre: declarar resultados, límites y evolución futura con enfoque crítico.
